# NetVis v0.5.0 — Format breadth (OpenVINO-first)

> Status: **planned**. **Resequenced 2026-07-21**: this milestone (native format
> parsers) lands *before* the plugin milestone (now `docs/v0.6.0-plan.md`), because
> the native parsers are the concrete reference implementations that de-risk the
> v0.6.0 `ParserPlugin` ABI freeze. OpenVINO IR is the headline: it is the format
> that most cleanly fits the zero-payload thesis (an XML topology + an
> offset-addressed weight blob), so it is implemented first and is the shape the
> rest of the milestone rhymes with.

## Goal

Widen the set of files NetVis opens without discarding a single invariant it has
defended since v0.1: **single self-contained binary**, **cross-platform**, **zero
tensor-payload reads** (record offset+len, never decode), **hostile-input-safe**
(bounds-checked, bounded nesting), **never throw / never crash on bad input**
(malformed → `Result` error with a byte offset, or a partial/labelled model).

Six formats, delivered in priority order, each closing a milestone issue:

| # | Format | Mode | New reader tech | Risk |
|---|---|---|---|---|
| **#39** | OpenVINO IR (`.xml`+`.bin`) | **graph** | hand-rolled XML-subset reader | **high** (new reader) |
| **#43** | ONNX external-data (`.onnx`+sibling) | graph | *none — already wired* | **low** (close a gap) |
| **#41** | NumPy `.npz` | tensor-table | miniz + `.npy` header parse | low |
| **#42** | Keras `.h5`/`.keras` | tensor-table | minimal HDF5 superblock walk | medium |
| **#38** | CoreML `.mlmodel` | graph | reuse `onnx::WireReader` | medium |
| **#40** | TorchScript archive | tensor-table + op list | extend `pytorch::parse_zip` | medium |

## What the codebase already gives us (confirmed by reading it)

The recon that makes this milestone mostly-additive rather than invasive:

- **CMake auto-discovers parsers.** The core library globs
  `src/parsers/*.cpp` with `CONFIGURE_DEPENDS` (root `CMakeLists.txt`). A new
  `src/parsers/<fmt>/Foo.cpp` is compiled with **zero CMake edits** — just a
  fresh configure. No per-parser build wiring.
- **The wire reader is reusable.** `parsers/onnx/WireReader.h` is a self-contained,
  bounds-checked, absolute-offset protobuf reader (`WireReader`, `SubRange`,
  `read_len_delim` records offset+len without touching bytes). CoreML (#38) is
  protobuf → it includes this header directly. **Not** ONNX-specific in substance,
  only in namespace.
- **Sibling-file mmap already works.** `TensorRef::external_path` (a `StringId`) +
  `engine/TensorStats.cpp::resolve_payload` already open a sibling file relative to
  `ModelSession::model_dir()` and bounds-check the range. This is the entire
  mechanism OpenVINO's `.bin` and ONNX external-data need — **it exists and is
  tested** (the ONNX fixture already carries an `external_data` initializer).
- **miniz is linked into core.** `parsers/pytorch/PytorchParser.cpp` already uses
  `mz_zip_reader_*` and — crucially — `payload_offset_from_local_header()` to turn a
  ZIP entry's local header into an **absolute mmap offset without extracting the
  payload**. `.npz` (#41), `.keras` (#42, it's a zip), and TorchScript (#40) all
  reuse this exact helper.
- **Tensor-table mode is free.** `has_graph = false` + `flat_tensors` already drives
  the virtualized `view/TensorTable.cpp` with dotted-key module hierarchy. Any
  graph-less format (npz, keras, torchscript-params) just populates `flat_tensors`.
- **The thesis is test-enforced.** `ByteReader::payload_read_counter()` is a
  thread-local counter; every parser test resets it to 0 and asserts it is still 0
  after parse. Every new parser gets the same assertion for free — it's the
  acceptance bar.
- **Fixtures are hand-encoded, stdlib-only.** `tools/gen_fixtures.py` emits every
  fixture byte-by-byte (no ML/protobuf libs). Each new format adds a `build_<fmt>()`.

## The one genuinely new piece: a hand-rolled XML-subset reader

OpenVINO IR topology is XML. NetVis has **no XML parser dependency** and will not
add one (single-binary invariant, and general XML parsers are a hostile-input
minefield). We build a **minimal, bounded, hostile-input-safe XML-subset reader**
in the same spirit as `WireReader`: it reads *structure* only, over the
bounds-checked byte range, and can never over-run.

**Scope of the subset (only what OpenVINO IR uses):**
- Elements, attributes (`name="value"`), nesting, self-closing tags (`<port … />`).
- Attribute values are the payload we care about; text nodes are ignored (IR has
  none that matter). CDATA is skipped structurally.
- Skip the XML declaration `<?xml …?>`, comments `<!-- … -->`, and any DOCTYPE.

**Hardening (non-negotiable, mirrors WireReader discipline):**
- **Bounded depth** (a `kMaxDepth` like ONNX's `kMaxGraphDepth = 64`) → adversarial
  nesting returns an error, never a stack overflow.
- **No entity expansion.** We do **not** implement `&entity;` expansion beyond the
  five built-ins as literal-ish passthrough → immune to the "billion laughs" /
  XXE class by construction (no DTD, no external entities, ever).
- Every read goes through `ByteReader`; truncation → `Result` error with byte
  offset. Unterminated tag / attribute → error, not a loop.
- Total, allocation-bounded: attribute count and name/value lengths are read into
  interned strings; a pathological file fails to parse, it does not hang or OOM.

Lives at `src/parsers/openvino/XmlReader.{h,cpp}`. **Frozen header first**, syntax-
checked in isolation, before the OpenVINO parser binds to it — same
freeze-contract-then-fan-out method as prior milestones. It is deliberately *not*
placed in `core/` yet: it is proven against OpenVINO first; promotion to a shared
reader is a later call once a second consumer exists.

## Shared spine (lead owns this, lands first — everything else forks from it)

Small, central, conflict-prone → **not fanned out**. One commit the lead makes and
freezes before any parser agent starts, so the parser TUs compile against a stable
surface:

1. **`parsers/Parser.h` — extend the `Format` enum** (append only; never renumber —
   detection and any persisted state key off names, not values):
   ```cpp
   enum class Format : uint8_t {
     Unknown, ONNX, TFLite, SafeTensors, GGUF, PyTorchZip, PyTorchLegacy,
     OpenVINO,   // .xml + sibling .bin
     Npz,        // NumPy zip archive
     Keras,      // .h5 / .keras (HDF5, or keras-v3 zip)
     CoreML,     // .mlmodel (protobuf)
   };
   ```
   Add the matching parser-namespace forward declarations:
   ```cpp
   namespace openvino { Result<ir::Model> parse(const MappedFile&, ProgressSink&); }
   namespace npz       { Result<ir::Model> parse(const MappedFile&, ProgressSink&); }
   namespace keras     { Result<ir::Model> parse(const MappedFile&, ProgressSink&); }
   namespace coreml    { Result<ir::Model> parse(const MappedFile&, ProgressSink&); }
   ```
   (TorchScript #40 rides the existing `pytorch::parse_zip` — no new namespace.)

2. **`parsers/Detect.cpp` — `format_name()` cases + detection.** The subtle part is
   **ZIP disambiguation**: `.npz`, `.keras` (v3), and PyTorch `.pt` all begin with
   the local-file-header magic `PK\x03\x04`. Content sniffing (not just extension):
   - **npz**: a ZIP whose first entries end in `.npy` (peek a central-dir/local
     filename). Extension `.npz` is the tiebreaker.
   - **keras v3**: a ZIP containing `config.json` + `metadata.json` (+
     `model.weights.h5`). Extension `.keras`.
   - **keras legacy / .h5**: HDF5 superblock magic `\x89HDF\r\n\x1a\n` at offset 0
     (or 512/1024/… — HDF5 allows the superblock at `2^k * 512`; we check offset 0
     first, then a bounded set of aligned offsets). Extension `.h5`/`.hdf5`.
   - **PyTorch zip**: contains `data.pkl` (existing logic — keep as the fallback so
     no regression).
   - **OpenVINO**: **not** binary-magic. XML text; sniff the first ~512 bytes for a
     `<net ` root element (optionally after `<?xml…?>`) carrying a `version="10"` or
     `version="11"` attribute. Extension `.xml` is a strong tiebreaker (bare `.xml`
     that isn't `<net` → `Unknown`, never a false positive).
   - **CoreML**: `.mlmodel` is a bare `Model` protobuf. Sniff like ONNX
     (`looks_like_onnx_proto`-style structural check) but keyed to CoreML's
     top-level fields; **extension `.mlmodel` is the decisive tiebreaker** because a
     bare protobuf is otherwise ambiguous with ONNX. Detection precedence: try ONNX
     structural first (it has the `ir_version`/`graph` signal); fall back to CoreML
     on the `.mlmodel` extension.
   - **`.mlpackage`** is a *directory* (a bundle), not a file → **out of scope for
     v0.5.0**; `.mlmodel` single-file only. Noted as a follow-up (directory-open is
     a `ModelSession`/`File→Open` change, not a parser change).

3. **`parsers/Detect.cpp::parse_model()` — dispatch** the four new `Format` values to
   their `parse()` entry points; `PyTorchZip` continues to route to
   `pytorch::parse_zip` (which #40 extends internally).

**Ordering rule** (learned from the v0.4.1 merges): the spine is one small commit;
the lead builds core-only + full doctest on it *before* branching the parser
agents, so every agent forks from a green, frozen surface. Detection precedence is
decided here, once, to avoid two agents both claiming `PK\x03\x04`.

## Per-format implementation

### #39 — OpenVINO IR (`.xml` + `.bin`) → **graph mode** *(headline)*

Source-verified against the OpenVINO IR v10/v11 schema.

**Files:** `src/parsers/openvino/XmlReader.{h,cpp}` (frozen first) +
`src/parsers/openvino/OpenVinoParser.cpp`.

**XML → IR mapping:**
- Root `<net name= version=>`. `<layers>` holds `<layer id= name= type=>` elements;
  `<edges>` holds `<edge from-layer= from-port= to-layer= to-port=>`.
- Each `<layer>` → one `ir::Node`. `type` → `op_type` (see alias table below).
  `<data …>` child attributes → `ir::Attribute`s (strides, pads, etc.).
- **Ports define the SSA edges.** A layer has `<input><port id= …>` and
  `<output><port id= precision=>` children, each with `<dim>` children giving the
  shape. We synthesize a stable value name per `(layer_id, port_id)` and wire
  `Graph::edge_refs` from `<edges>`: an edge `from (L,p) to (L',p')` connects
  layer L's output port p to layer L''s input port p'. `ValueInfo.shape` comes from
  the `<dim>` list; `ValueInfo.dtype` from the port `precision`.
- **Const weights → `TensorRef` with offset+len, zero payload read.** A `Const`
  layer's `<data>` carries `offset=` and `size=` (bytes into the sibling `.bin`) plus
  `element_type=`/`shape=`. Emit an initializer `TensorRef{ external_path=<bin>,
  file_offset=offset, byte_len=size, dtype, shape }`. **This reuses the ONNX
  external-data plumbing verbatim** — `resolve_payload` already opens `<model>.bin`
  relative to `model_dir`. The `.bin` sibling name is `<stem>.bin` (same stem as the
  `.xml`); record it as the `external_path`.
- **dtype normalization.** OpenVINO spells types lowercase in `element_type`
  (`f32`, `f16`, `bf16`, `i8`, `u8`, `i32`, `i64`, `boolean`, …) and UPPERCASE-ish in
  port `precision` (`FP32`, `FP16`, `I32`, …). One `map_ov_dtype()` handles both
  spellings → `ir::DType`. Unknown → `DType::Unknown` (honest).
- **op alias table.** OpenVINO op names differ from ONNX (`ReLU`→Activation,
  `Convolution`→Conv, `MatMul`, `Const`/`Constant`, `Parameter`=graph input,
  `Result`=graph output, `Add`/`Multiply`=Elementwise, …). A small
  `normalize_ov_op()` maps the common set so `OpCategory`/cost/color light up; an
  unmapped type passes through as-is and categorizes as `Other` (never a crash).
- **dynamic dims.** A `<dim>-1</dim>` or symbolic dim → `-1` (matches IR convention).
- **subgraph bodies.** `TensorIterator`/`Loop` carry a `<body>` (a nested `<net>`-like
  `<layers>`/`<edges>`); `If` carries `<then_body>`/`<else_body>`. Parse each as a
  nested graph appended to `Model::graphs` (reserve-slot-then-recurse, exactly like
  ONNX `parse_graph`), set `Node::subgraph`. Depth-bounded.
- **skip cleanly:** `<rt_info>`, `<meta_data>`, `<quantization_parameters>`, and any
  pre/post-process blocks — structurally skipped, never required.

**Detection:** first ~512 bytes contain `<net ` (after optional `<?xml?>`) with
`version="10|11"`. `.xml` extension is the tiebreaker.

**Fixture (`build_openvino()` in gen_fixtures.py):** emit a tiny `model.xml` +
`model.bin`. Graph: `Parameter → Convolution(Const weight) → ReLU → Result`. The
Const's `<data offset="0" size="16" element_type="f32" shape="2,2"/>` points into a
16-byte `model.bin`. The `.bin` is written as a **sibling file** so the external-data
resolution path is exercised end-to-end. Both files hand-written as text/bytes.

**Test (`tests/test_openvino.cpp`):** parse `model.xml`; assert node count + op
types (Convolution/ReLU present), that edges wired the ports (a value shape
resolved from `<dim>`s), that the Const initializer carries `external_path` ending
`.bin` + `byte_len==16` + `file_offset==0`, and — the invariant —
`payload_read_counter()==0`. A second test feeds a truncated `<net>` and asserts a
clean `Result` error (no crash). Plus a hostile-XML test: deeply nested tags hit the
depth cap and error.

### #43 — ONNX external-data → **graph mode** *(close a gap; already ~wired)*

The parser (`OnnxParser.cpp` case 13/14) already records `external_path` + offset +
length, and `resolve_payload` already opens the sibling relative to `model_dir`.
**What's missing is only proof + the multi-file open story:**
- **Fixture:** extend `build_onnx()` (or add a variant) so the `external_data`
  initializer points at a real sibling `weights.bin` that `gen_fixtures.py` also
  writes, with a *resolvable* offset (the current fixture uses a >4GB offset into a
  nonexistent file purely to test 64-bit carrying — keep that one, add a *small,
  real* sibling for the resolution test).
- **Test (`tests/test_onnx_external.cpp`):** parse; take the external initializer;
  call `resolve_payload(t, base, model_dir, &model)` and assert it opens the sibling
  and the bounds check passes — then assert `mark_payload_read` accounting is exactly
  the one intended read (i.e. structural parse still 0; the resolve is the explicit
  read). This makes #39's reuse of the same path a *tested* reuse.
- No parser code change expected; if a gap surfaces (e.g. relative-path join on
  Windows separators), fix it in `resolve_payload`. Small scope by design.

### #41 — NumPy `.npz` → **tensor-table mode**

**File:** `src/parsers/npz/NpzParser.cpp`.

- A `.npz` is a ZIP of `<name>.npy` entries (usually STORED/uncompressed). Reuse
  `mz_zip_reader_init_mem` + `payload_offset_from_local_header()` (lift the helper
  from `PytorchParser.cpp` — either share it or duplicate the ~15 lines; propose
  sharing via a tiny `parsers/ZipUtil.h` if the lead wants, else duplicate to keep
  agents disjoint).
- For each entry: read **only the `.npy` header** (magic `\x93NUMPY`, version, u16/u32
  header len, then the `{'descr':…, 'shape':(…)}` dict). Parse `descr` → `ir::DType`
  (invert the `npy_descr()` table already in `TensorStats.cpp`), parse `shape`. The
  payload offset = entry payload offset + `.npy` header length; record
  `TensorRef{ file_offset, byte_len, dtype, shape }`. **Header bytes are structural
  (counted as name/dtype reads, not payload); the array data is never touched.**
- Entry name (strip `.npy`) → tensor name; `has_graph=false`; push to `flat_tensors`.
- Compressed entries (rare for `.npz`, from `np.savez_compressed`): record
  shape/dtype from the header but set `file_offset=UINT64_MAX` (payload not directly
  mmap-addressable) + a metadata note, exactly like the legacy-pickle honest
  fallback. Never fake an offset.

**Detection:** ZIP whose entries are `.npy`; `.npz` extension tiebreaker.
**Fixture:** `build_npz()` — zip (STORED) two hand-written `.npy` arrays.
**Test:** two tensors, correct shapes/dtypes, real offsets, `payload_reads==0`.

### #42 — Keras `.h5` / `.keras` → **tensor-table mode**

**File:** `src/parsers/keras/KerasParser.cpp` + a minimal
`src/parsers/keras/Hdf5Reader.{h,cpp}`.

Two on-disk shapes under one `Format::Keras`:
- **`.keras` (Keras v3):** a ZIP containing `config.json`, `metadata.json`, and
  `model.weights.h5`. Detect by those entries; surface `config.json` (small, via the
  existing nlohmann/json dep) for layer names/metadata, and read weights from the
  embedded `model.weights.h5` via the HDF5 reader below (its payload offset within
  the outer zip is resolvable through the same local-header math when STORED; if the
  inner `.h5` is compressed, fall back to header-only/metadata like npz-compressed).
- **`.h5` / legacy `.keras`:** raw HDF5.

**Minimal HDF5 reader (the medium-risk part).** HDF5 is a complex format; we
implement only the **offset+len discovery** subset:
- Superblock magic `\x89HDF\r\n\x1a\n` at offset 0 (or an aligned `2^k*512`), read the
  version + the root group's object-header address and the size-of-offsets/lengths.
- Walk groups (symbol-table / B-tree + local heap, or the newer link-message form)
  to enumerate **datasets**; for each dataset read its **dataspace** (dims) and
  **datatype** (class+size → `ir::DType`) messages and its **data layout** message
  (contiguous → a single `(offset, size)` into the file). Record a `TensorRef` with
  that offset+len. **Chunked layout** (B-tree of chunks) → we do *not* stitch chunks
  in v0.5.0; record shape/dtype with `file_offset=UINT64_MAX` + a "chunked; payload
  not linearly addressable" metadata note (honest, no fake offset).
- Bounded everywhere (object-header message count cap, group-depth cap, refuse
  cyclic B-tree links via a visited set — same discipline as the pickle DAG guard).
  Any parse failure → partial/empty model + a note, never a crash.
- **This reader is scoped to structure discovery, not general HDF5 support.** That
  bound is what keeps it tractable and hostile-input-safe.

**Detection:** HDF5 magic for `.h5`; the config.json+weights.h5 zip shape for
`.keras`. **Fixture:** `build_keras_h5()` — hand-encode a *tiny* HDF5 file with one
contiguous dataset (this is the hardest fixture; if hand-encoding a full valid HDF5
proves too large for the fixture budget, ship a minimal superblock+one-dataset
file and document the exact byte layout in the generator, same as the TFLite
flatbuffer was hand-built). **Test:** dataset surfaced as a tensor with shape/dtype,
contiguous offset recorded, `payload_reads==0`; chunked-dataset variant asserts the
honest `UINT64_MAX` + note.

> If HDF5 hand-encoding balloons past a reasonable fixture budget, the fallback is
> to scope #42 to the `.keras` v3 zip (config.json + embedded weights) for the graph/
> metadata surface and mark raw-`.h5` chunked/contiguous weight offsets best-effort —
> decided during implementation, flagged to the lead, not silently narrowed.

### #38 — CoreML `.mlmodel` → **graph mode** (best-effort) / tensor-table fallback

**File:** `src/parsers/coreml/CoreMlParser.cpp`. **Reuses `onnx::WireReader`**
(include the header; it is not ONNX-specific in substance).

- `.mlmodel` is a protobuf `Model` (CoreML `Model.proto`). Top-level: `specification
  version` (field 1), `description` (field 2), and a `oneof Type` selecting the model
  kind (neuralNetwork / mlProgram / treeEnsemble / pipeline / …). Walk structurally
  with `WireReader`, recording only what we can map:
  - **neuralNetwork:** repeated `NeuralNetworkLayer`, each with a name, an input/output
    name list, and a `oneof` layer type → `ir::Node` (op_type = the layer-kind name),
    edges wired from the input/output name lists (same name→ValueInfo scheme as ONNX).
    Weights live in `WeightParams` (`floatValue`/`float16Value`/`rawValue` bytes) →
    record the `rawValue`/`quantized` **length-delimited sub-range offset+len** as a
    `TensorRef`, never decoded (identical to ONNX `raw_data` handling).
  - **mlProgram:** has weights in a **separate `weights/weight.bin`** when packaged as
    `.mlpackage` (out of scope, dir bundle) — for a bare `.mlmodel` mlProgram, record
    the structural ops best-effort and note weight residence.
  - Anything we don't map → node with the raw type name, categorizes as `Other`.
- If no recognizable graph, fall back to `has_graph=false` + whatever tensors we
  found (tensor-table), plus a metadata note — never error out on an exotic subtype.

**Detection:** structural protobuf sniff; **`.mlmodel` extension is the decisive
tiebreaker** (a bare protobuf is otherwise ambiguous with ONNX — ONNX structural
detection runs first, `.mlmodel` catches the rest). **Fixture:**
`build_coreml_mlmodel()` — a tiny hand-encoded `Model` protobuf with one
neuralNetwork layer + a `rawValue` weight (offset+len). **Test:** layer surfaced as a
node, weight `TensorRef` records offset+len, `payload_reads==0`; truncated-proto test
asserts clean error.

### #40 — TorchScript archive → **best-effort op listing** (extend pytorch)

**File:** extend `src/parsers/pytorch/PytorchParser.cpp` (no new namespace).

- A TorchScript archive is the same ZIP as a `.pt`, but additionally carries
  `constants.pkl` and a `code/` directory of serialized TorchScript source
  (`*.py`-like text). `parse_zip` **already detects** `has_constants`/`has_code` and
  emits a "graph view not supported" note. #40 upgrades that note into a **best-effort
  op/method listing**:
  - Enumerate `code/*.py`-style entries (structural — read the small text blobs, they
    are code, not tensor payload). Extract the top-level `def forward(...)` / method
    names and, best-effort, the `torch.*`/`ops.*` call op names via a **bounded,
    non-executing text scan** (regex-free simple tokenizer; we are not parsing
    Python, just surfacing identifiers after `torch.`/`aten::`/`prim::`). This is
    explicitly labelled best-effort/heuristic in the model metadata.
  - Surface these as either metadata entries or, if a clean node list is extractable,
    lightweight `flat_tensors`-adjacent info. **Full graph reconstruction stays a v1
    non-goal** (README already says so) — this is an op *inventory*, not a compute
    graph. Keep `has_graph=false`; the params table (existing behavior) plus an
    "ops seen" listing.
  - Parameters continue to surface via the existing `collect_tensors` state-dict
    walk. Zero payload reads unchanged (code blobs are structural text, weights stay
    offset+len).

**Detection:** unchanged (`data.pkl` in a `PK` zip → `PyTorchZip`); the code/constants
presence is detected *inside* `parse_zip`. **Fixture:** extend `build_pytorch()` to
add a `code/model.py` text entry + `constants.pkl`, so the best-effort path has
something to list. **Test:** archive parses, params still surface, the op-listing
metadata is populated, `payload_reads==0`, and the pickle VM allowlist is still
enforced on `constants.pkl`.

## Invariant ledger (how each format upholds them)

| Invariant | OpenVINO | ONNX-ext | npz | keras | CoreML | TorchScript |
|---|---|---|---|---|---|---|
| Zero payload reads | ✅ Const offset+len | ✅ (existing) | ✅ .npy header only | ✅ dataset offset+len | ✅ WeightParams sub-range | ✅ code=text, weights=offset |
| Single binary | ✅ hand XML reader | ✅ | ✅ miniz | ✅ hand HDF5 subset | ✅ WireReader | ✅ |
| Never crash on bad input | ✅ bounded XML + depth cap | ✅ | ✅ bounds-checked | ✅ bounded HDF5 walk | ✅ WireReader bounds | ✅ non-executing scan |
| Cross-platform | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| Honest unknowns | ✅ unmapped op=Other | ✅ | ✅ compressed=UINT64_MAX | ✅ chunked=UINT64_MAX | ✅ exotic subtype note | ✅ labelled best-effort |

## Sequencing & verification (freeze-contract-then-fan-out)

1. **Lead: shared spine** (Format enum, Parser.h namespaces, Detect.cpp
   disambiguation + dispatch). Commit, build core-only + full doctest **green**,
   freeze. Nothing else starts until this is in.
2. **Lead: freeze `openvino/XmlReader.h`** (the one new reader contract) and
   `g++ -fsyntax-only` a TU including it. This is the highest-risk interface; it is
   frozen before the OpenVINO parser binds to it.
3. **Fan out parsers to parallel agents, worktree-isolated** (each in its own git
   worktree → own branch → own PR, titled with the issue #). Files are disjoint by
   directory (`openvino/`, `npz/`, `keras/`, `coreml/`) + one shared touch each:
   `gen_fixtures.py` (a new `build_*` fn, append-only) and a new `tests/test_*.cpp`.
   The append-only fixture edits are the only overlap; lead merges them in order and
   regenerates fixtures once. TorchScript (#40) and ONNX-ext (#43) touch existing
   files (`PytorchParser.cpp`, onnx test) so they are sequenced to avoid colliding
   with each other, not parallel with themselves.
   - Priority/merge order: **#39 first** (headline, and it proves the external-data
     reuse), then #43 (locks that reuse with a test), then #41/#42/#38 in parallel,
     then #40.
4. **Lead: integration.** Regenerate all fixtures (`python3 tools/gen_fixtures.py
   tests/fixtures`), clean core-only build, **full** doctest (not just the new
   cases), release GUI build, then the three sanitizer presets (asan/ubsan/tsan).
   **Merge nothing red.** Each merge gated on the *merged-tree* build, not just the
   branch build (the lesson from the v0.4.1 `<cmath>` merge failure — verify the
   integration, not the island).
5. **Adversarial review sweep** (find → verify-to-refute), highest-value targets:
   - the XML reader on hostile input (unterminated tags, depth bomb, no-entity-XXE);
   - the HDF5 walk on cyclic B-tree links / bogus object-header addresses;
   - the ZIP disambiguation (a crafted zip that looks like npz *and* keras *and* pt —
     assert deterministic, documented precedence, no misroute);
   - the OpenVINO `.bin` offset/size bounds (a Const claiming `size` past the `.bin`
     end must error in `resolve_payload`, not over-read — it already bounds-checks;
     add the test).
   - **the thesis, per format:** `payload_read_counter()==0` after every parse.

## Non-goals (v0.5.0) — stated honestly

- **`.mlpackage` directory bundles** (CoreML) — directory-open is a session/UI change,
  not a parser change. `.mlmodel` single-file only this milestone.
- **HDF5 chunked-dataset payload stitching** — record shape/dtype, mark the payload
  non-linearly-addressable; do not fabricate an offset.
- **`np.savez_compressed` payload addressing** — same honest fallback (header known,
  offset UINT64_MAX).
- **Full TorchScript graph reconstruction** — remains the v1 non-goal (README §
  Non-goals); #40 is an op *inventory*, not a compute graph.
- **A general XML/HDF5 parser** — we implement only the bounded structural subset each
  format needs; general parsers are a dependency and a hostile-input surface we won't
  take on.
- **Nexus manifest companion view (#67)** — deferred to after these real parsers land;
  it is a workspace-index view, not a format parser (see #67).
