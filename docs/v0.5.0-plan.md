# NetVis v0.5.0 — Plugins (registry spine + declarative + WASM)

> Status: **planned** (drafted while v0.4.0 is in implementation). Depends on
> v0.4.0 landing first — v0.5.0's registry absorbs the op tables v0.4.0 finalizes
> (`OpCategory` table, `compute_flops` formula dispatch, `ShapeInferenceExt`
> handlers). This is the authoritative design; it ships as **three independently
> useful increments** (spine → declarative → WASM), not one mega-release.

## Goal

Let users and third parties extend NetVis without a NetVis release, without
breaking any of the invariants NetVis has defended since v0.1: **single self-
contained binary**, **cross-platform**, **zero tensor-payload reads**, **hostile-
input-safe**, **never throw / never crash on bad input**.

Concretely, four extension points, delivered in priority order:

1. **Op definitions** — category, color, FLOP formula, shape-propagation rule for
   an op the built-ins don't know (custom/exotic/domain ops). *The headline ask.*
2. **Format parsers** — new file formats (CoreML, OpenVINO IR, StableHLO, …).
3. **Analysis passes** — custom metrics / cost models over the parsed IR.
4. ~~View panels / exporters~~ — **explicit non-goal for v0.5.0** (needs native +
   ImGui context sharing or a retained-mode UI protocol; deferred, see Non-goals).

## Core architecture — one registry, three backends

The unifying idea: everything that answers "what is this op / how do I parse this
file / what does this pass compute" implements a small **frozen interface** and
**registers into one table**. Consumers (`CostModel`, `ShapeInferenceExt`,
`GraphCanvas`, `Detect`) ask the registry and never know which backend answered.

```
        ┌─────────────────────────────────────────────┐
        │              Plugin Registry                 │
        │  op handlers · parsers · passes (by key)      │
        └───────▲──────────────▲──────────────▲─────────┘
                │              │              │
        ┌───────┴───┐   ┌──────┴──────┐  ┌────┴────────┐
        │ Built-in  │   │ Declarative │  │   WASM      │
        │  (C++)    │   │ (TOML+DSL)  │  │  (wasm3)    │
        └───────────┘   └─────────────┘  └─────────────┘
   v0.4.0 tables →      manifest+formula    .wasm exports
   register here        → compiled closure  → marshalled calls
```

- **Built-ins register too.** The v0.4.0 `OpCategory` entries, FLOP formulas, and
  shape handlers are refactored to register into the registry at startup. This is
  the dogfood: the built-in path and the plugin path are the same path. It also
  makes increment 1 (the spine) provable by "all existing tests still pass" — pure
  internal refactor, zero behavior change.
- **Fallback ladder.** A lookup for an op resolves in precedence order: user WASM →
  user declarative → built-in → honest-unknown. (Precedence configurable; default
  lets a plugin *override* a built-in, e.g. to fix a wrong built-in FLOP formula,
  with a visible "overridden by plugin X" indicator.)
- **Backends compose on one op.** A declarative manifest can delegate just its
  shape rule to WASM (see Declarative §). Cheap parts stay declarative; the hard
  part escalates. Author pays the WASM cost only where unavoidable.

### Why this makes "both backends" cheap

The registry, plugin discovery, enable/disable UI, versioning, and trust/
provenance surface are built **once** and shared. The two backends are additive
registration paths into that spine, not two parallel systems:

| Component | Shared | Declarative-only | WASM-only |
|---|---|---|---|
| Registry + interfaces (`OpHandler`/`ParserPlugin`/`PassPlugin`) | ✅ | | |
| Discovery, manifest, enable/version/trust UI | ✅ | | |
| TOML loader | | ✅ | |
| Expression DSL (parse → closure) | | ✅ | |
| wasm3 embed + host API / marshalling | | | ✅ |

## The frozen interfaces (the plugin-era ABI — design carefully, version forever)

These are the contracts every backend binds to. Same header-freeze discipline as
v0.2–v0.4, higher stakes: declarative closures, WASM marshalling, AND built-ins
all implement them. **Freeze in the design phase; version with `api_version` and
refuse mismatches.** Kept pure (`ir/IR.h` + `core/` only) so op handlers stay
headless-testable exactly like `compute_flops`.

```cpp
// engine/plugin/OpHandler.h  (sketch — frozen surface)
namespace netvis {

// A read-only view of a node the handler may reason about WITHOUT touching a
// payload: op string, resolved input/output shapes+dtypes, and attributes. This
// is the whole vocabulary a FLOP/shape formula gets — deliberately no byte access.
struct OpContext {
  std::string_view op_type;                 // normalized (lowercased, domain-stripped)
  std::string_view domain;                  // original domain if any
  const SmallVec<int64_t,6>* input_shape(uint32_t slot) const;   // nullptr if absent/unresolved
  const SmallVec<int64_t,6>* output_shape(uint32_t slot) const;
  ir::DType input_dtype(uint32_t slot) const;
  ir::DType output_dtype(uint32_t slot) const;
  uint32_t  input_count() const, output_count() const;
  std::optional<int64_t>              attr_int(std::string_view) const;
  std::optional<std::string_view>     attr_string(std::string_view) const;
  const std::vector<int64_t>*         attr_ints(std::string_view) const;
  // NO method returns tensor bytes. Thesis is enforced by the vocabulary itself.
};

struct FlopResult  { uint64_t flops = 0; bool known = false; };   // honest-unknown
struct ShapeResult { /* per-output-slot shape+dtype, or "leave unknown" */ };

struct OpHandler {
  virtual OpCategory   category(const OpContext&)   const = 0;
  virtual Rgba8        color(const OpContext&)      const = 0;   // or "use category color"
  virtual FlopResult   flops(const OpContext&)      const = 0;
  virtual ShapeResult  infer_shape(const OpContext&) const = 0;
  virtual uint32_t     api_version()                const = 0;
};

}  // namespace netvis
```

Parser and pass interfaces (WASM-backed; see WASM §) are similarly frozen:

```cpp
struct ParserPlugin {                 // one whole file -> ir::Model, run ONCE
  bool can_parse(const uint8_t* head, size_t n) const;   // magic sniff, bounded
  ParseResult parse(HostFileView, ModelBuilder&) const;  // records offset+len only
  uint32_t api_version() const;
};
struct PassPlugin {                   // ir::Model -> named scalar/tensor metrics
  PassResult run(const ir::Model&, const CostReport&) const;  // pure, no payload
  uint32_t api_version() const;
};
```

> **Design-phase adversarial review MUST vet `OpContext`/`ShapeResult`** before any
> backend binds — the same "frozen contract → parallel fan-out" method as prior
> milestones. Getting the shape-result type and the attr/shape accessor set right
> is the whole ballgame; a later change is a plugin-ecosystem break.

## Increment 1 — Registry spine (internal refactor, no plugins yet)

- Add `engine/plugin/Registry.h` + `OpHandler.h`/`ParserPlugin.h`/`PassPlugin.h`.
- Refactor v0.4.0's built-ins to register a `BuiltinOpHandler` per op-family into
  the registry at startup (the `OpCategory` table, the `compute_flops` explicit
  dispatch, the `ShapeInferenceExt` handlers become registered handlers keyed by
  normalized op string). `Detect.cpp`'s format sniffers become registered
  `ParserPlugin`s.
- `CostModel`/`ShapeInferenceExt`/`GraphCanvas`/`Detect` call
  `registry.handler_for(op)` / `registry.parser_for(bytes)` instead of their
  hardcoded switches.
- **Acceptance:** every existing test passes unchanged, GUI byte-identical. This
  increment ships as a "no user-visible change" refactor whose value is enabling
  the next two. Bump `LayoutCache::kVersion`? No — nothing layout-affecting.
- **Perf gate:** `handler_for` is a hash lookup once per node, or hoisted per op-
  type; must not regress the shape-inference/cost hot path (benchmark against a
  v0.4.0 build on the decoder_model 3095-node graph).

## Increment 2 — Declarative backend (TOML manifest + expression DSL)

The low-floor path: no toolchain, no code, safe by construction. Covers ~90% of op
plugins (category/color/FLOP/simple shape rule).

### Manifest

```toml
api_version = 1
name = "my-ops"
author = "..."

[[op]]
name = "MultiHeadAttention"
domain = "com.microsoft"       # optional; matched after domain-strip too
category = "Attention"          # must be a known OpCategory
color = "#D864D0"               # optional; else category color
flops = "2 * (2*B*Sq*Skv*Hq)"   # expression; empty/omitted => flops unknown (honest)
[op.vars]                        # bind formula vars to shape dims / attrs
B  = "in0.shape[0]"
Sq = "in0.shape[1]"
Skv = "key.shape[1]"; key = "in1"
Hq = "in0.shape[2] * in0.shape[3]"
[op.shape]                       # per-output shape-propagation rule
out0 = "[B, Sq, in2.shape[-1]]"  # value shape as an expression list
# escalate ONE part to WASM when the DSL can't express it:
# shape.wasm = "myplugin.wasm:infer_myweirdop"
```

### Expression DSL (pure, total, no side effects)

- **Values:** `int64` (dims/attrs). `O` = output-0 element count; `in0.shape[k]`
  (negative k = from-end); `in0.rank`; attr names bound in `[op.vars]`.
- **Ops:** `+ - * / %` `min() max() prod()` `? :` (ternary), comparisons. `/` and
  `%` guard div-0 → the whole result is "unknown" (honest, never a fabricated
  number or a crash).
- **Compilation:** parse each expression to an AST/RPN **once** at load; eval per
  node with a `var → int64` environment. All arithmetic routes through the same
  saturating `safe_mul`/`safe_add` as the built-ins (a formula CANNOT overflow to a
  wrong small number). An unresolved var (missing shape/attr) → result unknown.
- **Safety:** the grammar has **no** file/memory/loop/recursion primitive — a
  malicious manifest can at worst fail to parse or yield a wrong number. Zero-
  payload is upheld because the vocabulary has no byte-access term. Fuzz the parser
  (bounded depth, reject pathological nesting) as part of the hostile-input suite.

### Loading / precedence
- Discovered from `plugins/` next to the binary + the user config dir; each op
  registers a `DeclarativeOpHandler` (compiled closures). A declarative handler may
  override a built-in (visible indicator) or add a new op.
- `api_version` mismatch → manifest rejected with a clear message, never loaded
  half-applied.

## Increment 3 — WASM backend (wasm3)

The high-ceiling path for what the DSL can't express: real parsers, stateful/
iterative shape logic, custom analysis passes. **wasm3** (single-file interpreter)
chosen over wasmtime to keep the single-binary story and portability; the perf
tradeoff is acceptable because **WASM stays on coarse entry points only.**

### Host API (imports the sandbox may call) — thesis enforcement lives here

The sandbox has no syscalls by default. NetVis exposes a **capability-scoped** host
API; the zero-payload thesis becomes a host-API design choice, not a convention:

```
// Parser plugin capabilities (parse runs ONCE per file):
host_file_len() -> u64
host_read_range(off, len, dst_ptr) -> ok   // host mediates; bounded; for MAGIC/
                                            // STRUCTURE bytes, not weight decode
host_record_tensor(name_ptr, off, len, dtype)  // declare a weight by OFFSET+LEN;
                                            // host stores TensorRef, NEVER hands the
                                            // decoded weight buffer to the view layer
host_add_value(...), host_add_node(...), host_intern(str) -> StringId
// Pass plugin capabilities: read-only IR handle accessors + emit named metric.
// There is NO host import that returns a decoded weight buffer to a plugin that
// could forward it to the view — the thesis is a property of the exposed set.
```

- **Marshalling:** data crosses as flat bytes in wasm linear memory; no C++ object
  crosses. Keep calls **coarse** (`parse(whole file)`, `run(whole pass)`), never
  per-node — the interpreter + copy cost makes fine-grained calls slow. A WASM op
  *shape* handler is allowed (escalation from declarative) but documented as the
  slow path; a WASM op called once per node in a 100k-node graph is a footgun the
  docs and a perf-guard warn against.
- **Sandbox guarantees:** no FS/network/memory outside the plugin's linear heap
  unless a host import grants it. Fuel/step limit + memory cap so a malicious or
  buggy `.wasm` can't hang or OOM the host (aborts → plugin disabled, app survives).
- **Authoring:** plugin authors compile Rust/Zig/C/AssemblyScript → `.wasm`.
  Portable: one `.wasm` runs on Win/Mac/Linux (vs native's per-arch/compiler
  binaries). We ship a thin SDK header + an example parser and example pass.

## Discovery, trust, versioning (shared spine, all increments)

- **Discovery:** `plugins/` beside the binary + user config dir (`layout_cache_dir()`
  sibling). Manifest lists name/author/api_version/capabilities.
- **Enable/disable UI:** a plugins panel — list discovered plugins, toggle, show
  provenance + which built-ins a plugin overrides. Persisted to `view_prefs.json`
  (per-key guarded, backward compatible — the established pattern).
- **Trust model:** declarative = safe by construction (load freely). WASM =
  sandboxed but still arbitrary logic → **disabled by default, explicit user
  enable per plugin**, with a clear "this runs sandboxed code" notice. Native =
  never (see Non-goals).
- **Versioning:** every plugin declares `api_version`; the host refuses mismatches
  with a legible message. The interfaces are the frozen ABI — treated like the
  header freezes: a breaking change is a major-version event with a migration note.

## Invariant ledger (how each is preserved)

| Invariant | Declarative | WASM | Native |
|---|---|---|---|
| Zero payload | ✅ DSL has no byte term | ✅ host API omits weight-buffer export | ❌ raw mmap access → **rejected** |
| Single binary | ✅ text files | ✅ wasm3 embeds; `.wasm` portable | ❌ per-platform |
| Never crash on bad input | ✅ parse-guard + saturating | ✅ fuel/mem cap, abort→disable | ❌ in-process crash |
| Hot-path perf | ✅ compiled closure | ⚠️ coarse-only (enforced) | ✅ |
| Cross-platform plugin artifact | ✅ | ✅ portable `.wasm` | ❌ |

## Non-goals (v0.5.0) — stated honestly

- **View panels / custom ImGui UI as plugins.** Needs either native (breaks
  thesis + single-binary + sandbox) or a retained-mode UI-description protocol
  over WASM (large, separate design). Deferred to a possible v0.6+; if ever done,
  panels are the *only* case that might justify native-behind-a-C-ABI-shim, and
  only for explicitly-trusted plugins.
- **Native `.so`/`.dll` plugins.** Rejected as a primary mechanism: no sandbox
  (arbitrary in-process code), unenforceable zero-payload (raw mmap access), C++
  ABI fragility across compiler/stdlib versions, per-arch/per-compiler artifacts
  that break the single-`.exe` distribution. Everything v0.5.0 needs is reachable
  via declarative + WASM without discarding an invariant.
- **Dequantization / payload-transforming plugins.** Out of scope by thesis.
- **Per-node WASM as a routine pattern.** Allowed as declarative-escalation but
  documented as the slow path; the fast per-node path is declarative closures.

## Sequencing & verification

Each increment is independently shippable and separately valuable:

1. **Registry spine** — internal refactor; acceptance = all v0.4.0 tests pass
   unchanged + no hot-path regression on the 3095-node decoder graph. (Mostly a
   v0.4.0 continuation: the tables already exist, they just move behind a registry.)
2. **Declarative backend** — manifest + DSL; first real plugins. Acceptance = a
   sample `my-ops.toml` adds a new op that colors, FLOP-counts, and shape-resolves;
   DSL parser passes the hostile-input/fuzz suite (bounded depth, div-0, overflow →
   saturating/unknown, never crash); overriding a built-in shows the indicator.
3. **wasm3 backend** — host API + example parser + example pass; sandbox fuel/mem
   caps proven (a hostile `.wasm` that loops/allocs is killed, app survives); a
   sample parser adds a toy format recording only offset+len (thesis holds:
   assert no host import can hand a weight buffer to the view).

Build/verify method unchanged from prior milestones: freeze the interface headers
first + `g++ -fsyntax-only` a TU including all of them, fan out the disjoint
implementation files to parallel agents, lead does the clean integration build,
then an adversarial review sweep (find → verify-to-refute) focused on: DSL parser
hostile input, WASM sandbox escape / fuel-limit / marshalling bounds, and the
thesis-enforcement property of the host-API set (the highest-value adversarial
target — prove no exposed import leaks payload bytes to a forwarding plugin).

## Open questions to settle in design

- **Manifest format:** TOML (human-friendly, comments) vs JSON (already have
  nlohmann for `view_prefs`). Leaning TOML for authoring ergonomics; would add a
  tiny TOML reader (or a minimal subset parser, hostile-input-safe) — weigh against
  reusing the existing JSON dep.
- **DSL surface:** exact operator/function set; whether to allow `let`-bindings
  beyond `[op.vars]`; how negative/`0`-dim (dynamic) propagates through a shape
  expression (proposal: any unresolved input dim → that output dim is `-1`/unknown,
  mirroring the built-in shape handlers).
- **Override precedence & safety:** should a declarative plugin be allowed to
  override a *built-in* FLOP formula silently, or require an explicit `override =
  true`? (Proposal: require `override = true` + show the indicator.)
- **wasm3 vs wasmtime escape hatch:** ship wasm3 only, or leave a compile-time
  option for wasmtime where speed matters? (Proposal: wasm3 only for v0.5.0; keep
  the host-API boundary runtime-agnostic so wasmtime is a later drop-in.)
- **SDK distribution:** how authors get the `.wasm` toolchain + our SDK header +
  the ABI docs; ship examples in-repo under `plugins/examples/`.
